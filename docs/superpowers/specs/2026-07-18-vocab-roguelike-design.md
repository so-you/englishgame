# 英语爬塔（English Roguelike）最终设计文档 v3

日期：2026-07-18

状态：**最终版，可进入实现计划**

平台：Web 浏览器（验证期），后续可封装为桌面端或移动端

## 1. 产品定位

英语爬塔是一款面向中文英语学习者的卡牌 Roguelike 游戏。它借鉴《杀戮尖塔》的回合制战斗、敌人意图、牌组构筑和路线选择，但长期成长来自真实的英语学习进度。

### 1.1 产品目标

- **游戏为体，学习为核**：即使去掉学习主题，战斗仍应具备明确决策；学习行为同时产生游戏收益。
- **先教后练，练后再测**：玩家不会因为尚未学过的内容受到惩罚。
- **支持多阶段内容**：完整产品支持初中、高中、大学和自定义词表；MVP 只发布一个 CET4 内置包。
- **词汇与语法共用学习基础设施**：两者拥有不同练习形式，但共享内容包、调度、日志和掌握度体系。
- **错误是反馈，不是羞辱**：答错不使卡牌失效，不造成额外生命惩罚，也不清空全部学习进度。
- **学习算法隐形运行**：玩家看到的是“本局正好出现需要复习的内容”，不需要理解 SRS 参数。

### 1.2 非目标

- 不把游戏做成带战斗动画的强制题库。
- 不承诺仅靠一次游玩即可长期掌握单词或语法。
- 不在 MVP 中复刻《杀戮尖塔》的全部地图、职业、事件和卡牌规模。
- 不在浏览器包中保存第三方 AI 服务密钥。

## 2. 核心设计决策

### 2.1 双层卡牌模型

战斗能力与学习内容必须解耦。一张战斗卡由两层组成：

1. **卡牌躯壳（CardTemplate）**：费用、攻击、格挡、抽牌、状态效果等纯游戏能力。
2. **学习铭文（LearningUnit）**：本次学习会话绑定的单词词义或语法点。

两者在一局开始时绑定为 `CombatCardInstance`，本局内保持稳定。跨局时，SRS 调度器可以把新的到期内容绑定到玩家的卡牌躯壳上。

这一设计确保：

- 学习项目掌握度不会削弱卡牌基础战力。
- 单词毕业不会删除玩家辛苦构筑的战斗卡。
- 卡牌平衡与学习算法可以独立测试和迭代。
- 自定义词汇不需要自动生成新的战斗技能。
- 语法题可以使用同一套战斗牌组，而不需要伪装成“单词卡”。

### 2.2 概念映射

| 卡牌 Roguelike 概念 | 本作设计 |
|---|---|
| 卡牌 | 卡牌躯壳 + 本局学习铭文 |
| 出牌 | 基础效果始终生效，可消耗专注进行过载回忆 |
| 卡牌升级 | 本局战斗能力升级，与学习掌握度无关 |
| 牌组构筑 | 选择战斗能力组合；学习内容由调度器填充 |
| 长期成长 | 学习项目掌握度、毕业墙、内容包进度 |
| 删除卡牌 | 只影响本局战斗构筑，不等同于删除学习内容 |
| 遗物 | 改变战斗规则或学习交互，但不能篡改历史答题结果 |
| 死亡 | 结束本局；已经发生的有效学习记录照常保存 |

## 3. 核心体验流程

### 3.1 MVP 的一次学习会话

MVP 中，一次会话是一条固定的线性流程，包含两场普通战和一场 Boss 战，目标时长 10–15 分钟。

1. **选择内容包**：MVP 默认 CET4；用户也可选择已导入的自定义词表。
2. **生成学习集合**：调度器优先选择最多 8 个到期项目，再加入最多 4 个新项目；空位由未到期练习项目填充。
3. **战前学习新内容**：新项目在安全的准备界面完成首次教学，不在战斗中弹出完整教学面板。
4. **生成战斗牌组**：12 张固定卡牌躯壳与本次学习集合一对一绑定；学习项目不足时，剩余躯壳作为不能过载的中性卡，不用重复内容强行填满。
5. **进行三场战斗**：玩家自由出牌，并选择是否消耗专注进行过载。
6. **局间奖励**：普通战后提供回血或遗物选择，不在 MVP 中新增学习项目。
7. **会话结算**：展示本局尝试、到期项目覆盖率、易错项和下次复习时间，不用“考试分数”羞辱玩家。

### 3.2 完整版爬塔循环

MVP 验证通过后，扩展为：

1. 选择角色、内容包和新内容配额。
2. SRS 生成本局学习集合，玩家可预览但不能通过反复刷新规避到期内容。
3. 程序生成分支地图：战斗、精英、篝火、商店、事件和 Boss。
4. 战后选择卡牌躯壳或本局升级；新学习项目只在奖励界面完成教学后进入后续战斗。
5. 死亡或通关后保存真实回忆日志，并结算本局构筑与长期学习进度。

## 4. 学习领域模型

### 4.1 内容分级

```ts
type EducationStage = 'junior' | 'senior' | 'university' | 'custom'

type CurriculumTag =
  | 'junior'
  | 'senior'
  | 'cet4'
  | 'cet6'
  | 'postgraduate'
  | 'custom'
```

`EducationStage` 用于产品入口和内容包分类；`CurriculumTag` 用于更细的课程或考试标记，两者不得混用。

### 4.2 原子学习项目

一个 `LearningUnit` 必须代表一个可以独立调度和判断掌握度的目标。

- 词汇按**目标词义**拆分。同一个单词的不同词义可以拥有不同 `id` 和 SRS 状态。
- 语法按**语法概念**调度，但必须轮换多个练习，不能反复考同一句话。

```ts
type LearningUnit = VocabularyUnit | GrammarUnit

interface SourceInfo {
  kind: 'builtin' | 'import' | 'llm-enriched'
  name?: string
  license?: string
  importedAt?: number
}

interface VocabularyUnit {
  id: string
  type: 'vocab'
  term: string                    // 支持单词和短语
  normalizedTerm: string
  senseId: string                 // 一个 unit 对应一个目标词义
  definitionZh: string
  acceptedMeanings?: string[]
  phonetic?: string
  partOfSpeech?: string
  example?: string
  exampleZh?: string
  roots?: string
  curriculumTags: CurriculumTag[]
  frequencyRank?: number
  enrichmentStatus: 'complete' | 'pending' | 'incomplete'
  source: SourceInfo
}

interface GrammarUnit {
  id: string
  type: 'grammar'
  name: string
  explanationZh: string
  examples: Array<{ en: string; zh: string }>
  exerciseIds: string[]           // 内置内容至少 4 道不同练习
  curriculumTags: CurriculumTag[]
  source: SourceInfo
}
```

### 4.3 练习与答案

```ts
type ExerciseKind =
  | 'vocab-meaning-choice'
  | 'vocab-word-choice'
  | 'vocab-spelling'
  | 'vocab-listening-spelling'
  | 'grammar-cloze'
  | 'grammar-reorder'
  | 'grammar-error-correction'
  | 'grammar-transformation'

type AnswerSpec =
  | { kind: 'choice'; correctOptionId: string }
  | { kind: 'text'; accepted: string[]; caseSensitive?: boolean }
  | { kind: 'token-order'; acceptedOrders: string[][] }

interface Exercise {
  id: string
  unitId: string
  kind: ExerciseKind
  prompt: string
  options?: Array<{ id: string; text: string }>
  answer: AnswerSpec
  explanation: string
  difficulty: 1 | 2 | 3 | 4
}

interface ContentPack {
  id: string
  schemaVersion: number
  name: string
  stage: EducationStage
  curriculumTags: CurriculumTag[]
  units: LearningUnit[]
  exercises: Exercise[]
  source: SourceInfo
}
```

答案比较统一执行 Unicode NFC、首尾空白清理和空白折叠。拼写默认忽略大小写，但不自动接受会改变词义的模糊匹配；可能的轻微拼写错误可以提示用户确认，不直接判对。

## 5. 首次教学与练习进阶

### 5.1 三阶段学习闭环

1. **编码（Teaching）**：展示目标词义或语法解释、发音、例句和必要提示；不计分、不扣资源。
2. **有支架练习（Guided Practice）**：同一会话中使用低难度练习帮助提取；可获得战斗奖励，但 `graded=false`。
3. **到期回忆（Scheduled Recall）**：只有达到 `dueAt` 后的首次符合条件作答可以推进或降低 SRS 状态。

### 5.2 教学发生的位置

- MVP 的新项目统一在战前准备界面教学。
- 完整版新项目统一在战后奖励、商店或事件的安全界面教学。
- 战斗中不弹出包含大量说明的首次教学面板。
- 教学完成后，新项目从 `new` 转为 `learning`，初始掌握度为 L1，并安排次日首次复习。

### 5.3 题型 progression

#### 词汇

| 掌握度 | 默认题型 | 目标 |
|---|---|---|
| L0 | 首次教学 | 编码，不计分 |
| L1 | 看词选义 | 基础再认 |
| L2 | 看义选词 | 反向再认 |
| L3 | 看义拼词 | 主动提取与拼写 |
| L4 | 听音拼词或看义拼词 | 更少线索的主动提取 |
| L5 | 维护性复习 | 长期保持 |

#### 语法

| 掌握度 | 默认题型 | 约束 |
|---|---|---|
| L0 | 规则说明 + 正反例 | 不计分 |
| L1 | 选项式填空 | 使用新句子 |
| L2 | 无选项填空或排序 | 使用新句子 |
| L3 | 改错 | 不重复近期题目 |
| L4 | 句型转换 | 接受预先定义的合理答案 |
| L5 | 混合维护复习 | 从未近期使用的练习中抽取 |

每个语法概念的 SRS 状态属于概念本身，但计分练习必须轮换。一个练习在其他可用练习尚未出现前，不得连续两次作为计分题。

## 6. SRS 状态机

### 6.1 MVP 状态模型

MVP 使用可解释、可测试的分级调度，不同时维护容易漂移的 `mastery`、`stability` 和 `difficulty`。未来可以在不改变战斗接口的情况下替换为 FSRS 等算法。

```ts
type LearningStatus = 'new' | 'learning' | 'review' | 'maintenance'
type Mastery = 0 | 1 | 2 | 3 | 4 | 5
type RecallGrade = 'again' | 'hard' | 'good'

interface SrsState {
  unitId: string
  mastery: Mastery
  dueAt?: number
  intervalDays: number
  introducedAt?: number
  lastReviewedAt?: number
  lastGradedDueAt?: number
  consecutiveCorrect: number
  lapses: number
}

interface RecallLog {
  id: string
  unitId: string
  exerciseId?: string
  sessionId: string
  encounterId?: string
  scheduledDueAt?: number
  kind: 'teaching' | ExerciseKind
  correct?: boolean
  usedHint: boolean
  responseMs?: number
  eligible: boolean
  graded: boolean
  grade?: RecallGrade
  createdAt: number
}
```

### 6.2 明确状态转换

#### 首次教学

```text
new/L0
  --完成教学，不计分-->
learning/L1，intervalDays=1，dueAt=次日
```

教学日志永远是 `eligible=false, graded=false`。

#### 计分资格

一次尝试必须同时满足以下条件才有资格计分：

1. 项目已经完成教学。
2. `now >= dueAt`。
3. 当前 `dueAt` 尚未被计分，即 `lastGradedDueAt !== dueAt`。
4. 本次练习符合当前掌握度要求，且没有直接展示答案。

计分资格由 SRS 模块计算，战斗 UI 只能提交作答事实，不能自行指定 `graded=true`。

#### 计分结果

| 结果 | 状态变化 |
|---|---|
| 正确、无提示 | `good`：掌握度 +1（最高 L5），`consecutiveCorrect+1`，采用新等级对应间隔 |
| 正确但使用提示 | `hard`：掌握度不变，保持当前间隔，`consecutiveCorrect` 不增加 |
| 错误 | `again`：`lapses+1`，`consecutiveCorrect=0`，掌握度 -1（最低 L1），间隔改为 1 天 |
| 直接出牌、未尝试 | 不记录答题，不改变 `dueAt`，项目保持到期 |
| 到期前练习 | 记录为 `graded=false`，不改变调度 |
| 同一到期窗口后续练习 | 记录为 `graded=false`，不重复改变调度 |

MVP 的等级与间隔采用明确映射：完成教学进入 L1，间隔 1 天；`good` 后进入 L2/L3/L4/L5 时，间隔分别为 3/7/15/30 天；L5 再次获得 `good` 时保持 L5，间隔 30 天。`hard` 使用当前等级原有间隔重新安排；`again` 只降低一级并在次日重试，因此 L5 答错会进入 L4，而不是重置为新词。

状态由等级唯一派生：L0=`new`，L1–L2=`learning`，L3–L4=`review`，L5=`maintenance`。每次计分后以计分前的 `dueAt` 写入 `lastGradedDueAt`，更新 `lastReviewedAt`，再根据结果生成新的 `dueAt`。间隔数值集中配置，不写死在 UI。

### 6.3 防刷规则

- 新开一局会产生新的 `sessionId`，但不会产生新的到期窗口。
- `lastGradedDueAt` 使用计分前的 `scheduledDueAt`，确保反复开局不能刷等级。
- 同一项目在同一会话可反复用于战斗奖励，但最多一次影响 SRS。
- 遗物只能影响提示、战斗奖励和练习机会，不能把错误日志改成正确，也不能绕过到期资格。
- 死亡不会额外降低掌握度，只保留真实发生过的计分结果。

### 6.4 调度优先级

调度器按以下顺序生成学习集合：

1. 选择 `dueAt <= now` 的项目，先按 `dueAt` 从早到晚，再按掌握度从低到高、`lapses` 从高到低排序，最多取 8 个。
2. 按玩家设置的新内容配额加入 0–4 个 `new/L0` 项目；默认配额为 4。
3. 空位从最近学习但尚未到期的项目中补齐，这些项目只能练习，不能计分。
4. 一个学习项目在同一会话只绑定一张卡牌；候选数量不足时，剩余卡牌保持为不含学习铭文的中性卡。
5. 本次会话的集合生成后立即保存；重新加载、恢复会话或反复进入预览不得重新抽取。

超过 8 个的到期项目保持原 `dueAt`，在后续会话继续优先出现。调度器不得为了凑满新内容配额而挤掉已经到期的项目。

## 7. 战斗系统

### 7.1 基础规则

MVP 初始参数：

- 玩家初始生命 50，生命在三场战斗之间保留。
- 每回合获得 3 能量，摸 5 张牌，手牌上限 10。
- 回合结束弃掉手牌；抽牌堆为空时洗入弃牌堆。
- 卡牌费用为 0–2，基础效果包括攻击、格挡、抽牌和施加状态。
- 格挡在其拥有者下一个回合开始时清零；敌人生命降为 0 时战斗胜利，玩家生命降为 0 时本次会话结束。
- 敌人提前显示下一回合意图。
- 战斗失败结束本次会话，但已经提交的回忆日志照常保存。
- 战斗随机数由 seed 驱动，核心逻辑支持回放和确定性测试。

以上参数用于首轮试玩，不是不可更改的产品承诺。

### 7.2 卡牌数据

```ts
interface CardTemplate {
  id: string
  name: string
  cost: 0 | 1 | 2
  rarity: 'starter' | 'common' | 'uncommon' | 'rare'
  effects: CombatEffect[]
  tags: string[]
}

interface CombatCardInstance {
  id: string
  templateId: string
  learningUnitId?: string
  sessionId: string
  upgraded: boolean
}
```

没有 `learningUnitId` 的中性卡只能直接打出，不能过载。MVP 使用 12 张固定卡牌躯壳：5 张攻击、5 张防御、2 张功能牌。v0.2 再开放完整构筑、删牌和本局升级。

### 7.3 直接出牌

- 卡牌基础效果与学习掌握度完全无关，始终全额生效。
- 直接出牌不弹题、不计错、不改变 SRS。
- 玩家可以在状态不佳时只玩战斗，而不会被强迫接受测试。

### 7.4 专注与过载

- 每场战斗开始时获得 3 点专注，而不是每回合自动刷新。
- 每次过载消耗 1 点专注；没有使用的专注不会造成惩罚。
- 部分卡牌或遗物可以恢复专注，但 MVP 中恢复次数必须受限。
- 同一张卡直接打出不消耗专注。

过载结果：

| 情况 | 战斗结果 | 学习结果 |
|---|---|---|
| 正确 | 基础效果 + 题型对应加成，语流 +1 | 按 SRS 资格决定是否计分 |
| 错误 | 基础效果照常生效，无过载加成，语流最多 -1 | 展示答案与简短解释；符合资格时记 `again` |
| 使用提示后正确 | 基础效果 + 较小加成 | 符合资格时记 `hard` |
| 超出建议答题时间 | 不判失败，不扣战斗资源 | 只记录响应时间 |

L1–L4 的建议初始加成依次提高，以补偿题型难度。具体倍率由试玩平衡，不进入 SRS 逻辑。

### 7.5 防止玩家只挑会的词

“有限专注”只能控制题目数量，不能独自解决避难就易。因此增加以下机制：

- **勇气奖励**：每个到期或低掌握度项目，每场战斗第一次主动尝试时立即获得少量格挡；无论答案正确与否都生效，但同项目不能重复领取。
- **挑战加成**：低掌握度且到期的项目答对后，比未到期的熟练项目获得更高的额外收益。
- **覆盖率反馈**：局后展示“到期项目尝试覆盖率”，但不把未尝试标成错误。
- **调度不逃逸**：未尝试的到期项目保持到期，下一局仍优先出现；反复刷新候选池不会移除它。

新项目在战前完成教学后以 L1 进入战斗，可以进行有支架过载并获得正常战斗加成。因此新内容与熟悉内容拥有相同基础战力，也不会失去全部过载收益。

### 7.6 语流

- 过载成功后语流 +1，错误时最多 -1，不清空全部进度。
- 达到 3 点语流后自动消耗，下回合额外获得 1 能量。
- 提示后正确可以获得语流，但不获得额外速度奖励。
- MVP 只记录响应时间，不根据速度给予战斗优势，避免鼓励猜测或伤害无障碍体验。

### 7.7 MVP 战斗内容

以下内容是首轮可玩性测试的初始夹具，数值统一放入配置文件：

#### 初始牌组

| 数量 | 卡牌躯壳 | 费用 | 效果 |
|---:|---|---:|---|
| 4 | 打击 | 1 | 造成 6 点伤害 |
| 4 | 防御 | 1 | 获得 5 点格挡 |
| 1 | 重击 | 2 | 造成 12 点伤害 |
| 1 | 固守 | 2 | 获得 11 点格挡 |
| 1 | 洞察 | 1 | 抽 2 张牌 |
| 1 | 干扰 | 1 | 给予敌人 1 回合虚弱 |

虚弱使目标造成的伤害降低 25%。MVP 不需要更多状态；新增状态必须先定义叠加、持续时间和结算顺序。

#### 敌人

| 敌人 | 生命 | 意图循环 |
|---|---:|---|
| 墨团 | 28 | 造成 6 点伤害 → 获得 5 点格挡并造成 4 点伤害 |
| 回声蝠 | 32 | 连击 3×2 → 造成 8 点伤害 |
| Boss：遗忘守卫 | 85 | 造成 10 点伤害 → 获得 12 点格挡并施加 1 回合虚弱 → 连击 6×2 |

敌人只能改变战斗状态，不能打乱题目字母、缩短答题时间、替换正确答案或禁用无障碍设置。否则同一道题的计分条件不再一致，SRS 数据会失真。

#### 遗物

| 遗物 | 效果 |
|---|---|
| 词根罗盘 | 每场第一次过载正确后抽 1 张牌 |
| 错题本 | 每场第一次尝试到期项目时，勇气奖励额外获得 2 点格挡 |
| 语流护符 | 每场战斗以 1 点语流开始 |

第一场普通战后展示三个遗物供玩家三选一；第二场普通战后只提供回血，避免 MVP 奖励系统继续膨胀。遗物不得修改 RecallLog 的正确性、计分资格或历史记录。

## 8. 内容包与自定义词表

### 8.1 内置内容包

- 初中、高中、CET4、CET6 等内置包以课程标准和考试大纲的人工标签为主。
- 词频只用于包内排序和自定义内容兜底，不代替课程分级。
- 每条释义、例句、音标、音频和词频数据必须记录来源及许可。
- 内置语法概念至少准备 4 道不同练习；上线前检查答案唯一性和解释质量。

### 8.2 自定义导入范围

首版自定义导入只支持词汇和短语，不支持用户导入语法题。语法导入需要更复杂的答案校验，放到后续版本。

支持 CSV 和 TSV：

- 使用标准引号规则，允许释义和例句包含逗号。
- 默认 UTF-8，兼容 UTF-8 BOM；其他编码必须在预览页明确提示。
- 必填列：`term`。
- 可选列：`definition`, `phonetic`, `partOfSpeech`, `example`, `exampleZh`, `tags`。
- 提供模板下载，并允许用户在向导中完成列映射。

### 8.3 导入流程

1. 选择文件或粘贴表格文本。
2. 解析表头并映射列。
3. 预览前 10 行，同时展示检测到的编码、分隔符和错误。
4. 执行 trim、Unicode NFC、空白折叠和用于匹配的大小写归一；展示文本保留原始大小写。
5. 按 `normalizedTerm + definition` 去重。同词不同义保留为不同学习项目。
6. 使用有明确许可的本地词典匹配缺失释义、音标和例句；出现多个候选词义时必须让用户选择，不能自动合并。
7. 未匹配且没有释义的项目标记为 `pending`，不能进入战斗；用户补充释义后方可启用。
8. 提交前显示新增、更新、跳过和错误数量；确认后以单次事务写入 IndexedDB。
9. 生成含行号与原因的错误报告。下载为 CSV 时转义可能被表格软件解释为公式的单元格。

MVP 默认限制单次 5 MB、最多 5,000 行，避免浏览器页面被异常文件阻塞；限制应集中配置。

导入文件的一行默认代表一个目标词义。同一个词的多个词义应使用多行，或在预览界面由用户明确拆分；系统不能仅凭分号或斜杠自动推断词义边界。

### 8.4 富化与插画

- MVP 不调用 LLM，也不生成 AI 插画；排版卡必须在无插画条件下成立。
- 完整版内置包可由开发者离线批量生成插画，密钥不进入产品包。
- 自定义内容的 AI 富化只能使用后端代理或明确的 BYOK 模式。
- BYOK 密钥不得写入日志、导出文件或同步数据，默认只存本机安全存储；纯 Web 环境无法保证安全时不提供持久保存。

## 9. 节奏、反馈与无障碍

### 9.1 节奏硬约束

- 单场战斗目标 3–5 分钟，完整 MVP 会话 10–15 分钟。
- 每场最多 3 次常规过载，完整会话最多约 9 次；同一到期窗口最多一次影响 SRS。
- 每次学习交互尽量在当前战斗界面内完成，不跳转页面，不使用全屏考试弹窗。
- 4 选 1、拼写和听音只设置建议时间，不使用硬倒计时。
- 展示答案与解释后，玩家主动确认再返回战斗，避免信息一闪而过。

### 9.2 无障碍要求

- 支持全键盘操作、清晰焦点顺序和可见焦点状态。
- 支持关闭动画、减少位移效果和关闭建议计时提示。
- 不仅用颜色区分正确、错误、费用和敌人意图。
- 听力题必须提供等强度的非听觉替代题，例如看义拼词；不能直接显示会泄露答案的音标后仍按听力题计分。
- Web Speech API 不可用、语音缺失或播放失败时，自动切换到可计分的替代练习，不阻塞出牌。
- 中文解释、英文题干和音标应允许浏览器缩放与响应式换行。
- 错误文案描述答案和原因，不使用嘲讽、羞辱或连续失败惩罚动画。

## 10. MVP v0.1 范围

### 10.1 要验证的假设

1. 消耗有限专注进行过载，能够形成有趣而不是烦躁的战斗决策。
2. 玩家愿意主动尝试到期或不熟悉的内容，而不只挑会的词。
3. 一次 10–15 分钟会话能产生可观察的 24 小时和 7 天保持信号。

### 10.2 包含

- 一条固定线性流程：2 场普通战 + 1 场 Boss 战。
- 12 张固定卡牌躯壳，无完整牌组构筑。
- 30 个 CET4 高频词义单元。
- 2 种普通敌人、1 个 Boss、3 个可选遗物。
- 战前教学、有支架练习、过载、专注、勇气奖励和语流。
- 完整的 MVP SRS 状态机、防重复计分和回忆日志。
- CSV/TSV 自定义词表导入向导及本地词典匹配。
- IndexedDB 本地存档、进度重置和诊断数据导出。
- 键盘操作、减少动画、计时提示开关和听力替代题。

### 10.3 不包含

- 程序生成地图、商店、随机事件和完整构筑系统。
- 可游玩的语法内容；但核心数据类型和 SRS 日志必须能表达语法练习。
- 初中、高中、CET6 内容包 UI。
- AI 插画、LLM 富化、账号、云同步和排行榜。
- 移动端专项适配和原生应用封装。

## 11. MVP 验收标准

### 11.1 测试样本

- 至少邀请 10 名属于目标用户的英语学习者，而不只由开发者本人试玩。
- 每名参与者通过基线测试选择至少 8 个确实不会或不稳定的目标词义。
- 24 小时与 7 天测试使用未显示答案的独立任务，并保持题目口径一致。
- 小样本结果只用于产品 go/no-go，不用于对外宣称教育效果。

### 11.2 游戏性指标

| 指标 | 定义 | 初始目标 |
|---|---|---|
| 会话完成率 | 完成三场战斗的有效会话 / 开始的有效会话 | ≥ 70% |
| 专注使用率 | 已消耗专注 / 实际可用专注 | ≥ 50% |
| 到期项目尝试覆盖率 | 至少主动尝试一次的到期项目 / 本局出现的到期项目 | ≥ 60% |
| 次日再次游玩率 | 24–48 小时内再次开始有效会话的参与者 / 可回访参与者 | ≥ 40% |

需同时观察平均回合时长、每场过载次数、直接出牌比例和中途退出位置。过载使用率高但到期覆盖率低，视为机制没有完成学习目标。

### 11.3 学习性指标

- 基线测试分别记录“看到英文能写出核心中文义”和“看到中文义能拼出英文”。
- 24 小时目标：认义正确率 ≥ 70%，拼写正确率 ≥ 50%。
- 7 天结果作为保持趋势记录，不因小样本设置强制上线门槛；若较 24 小时大幅下降，优先修正调度和教学，而不是增加惩罚。
- 再认四选一只能作为辅助数据，不得替代独立认义或拼写测试。
- 语法版本上线前必须另行定义迁移题测试，不能用固定题重复正确率代替语法掌握。

## 12. 技术架构

### 12.1 技术栈

- Vite + React + TypeScript。
- Zustand 管理 UI 会话状态，framer-motion 仅负责可关闭的动画。
- Vitest 测试纯 TypeScript 领域核心。
- IndexedDB 保存内容包、SRS 状态、回忆日志、设置和会话快照。
- Web Speech API 作为 MVP 发音能力，必须有不可用检测和替代练习。

### 12.2 模块边界

```text
src/
  core/
    battle/       # 回合、卡牌、敌人、效果、确定性随机
    session/      # 学习集合、卡牌绑定、会话流程
    srs/          # 计分资格、状态转换、调度
    learning/     # LearningUnit、Exercise、答案判断
    content/      # 内容包、校验、版本迁移
    import/       # CSV/TSV 解析、映射、标准化、报告
  ui/
    battle/
    teaching/
    import/
    progress/
    settings/
  data/
    packs/
  infra/
    indexeddb/
    speech/
  tools/
    content-validation/
    offline-enrichment/
```

`src/core/` 不依赖 React、浏览器 DOM 或 IndexedDB，所有时间和随机数通过接口注入。

### 12.3 持久化要求

- 所有数据包含 `schemaVersion`，升级时执行可测试的迁移。
- SRS 状态与 RecallLog 分开保存；日志追加写，状态更新与计分日志使用同一事务。
- 支持导出学习进度和清空本地数据。
- 会话中断时保存快照；恢复后沿用同一个 `sessionId`，防止重新计分。
- 不在前端源码、IndexedDB、日志或错误报告中保存服务端密钥。

### 12.4 核心自动化验收

实现计划必须覆盖以下测试：

- 教学将 L0 转为 L1，但不产生计分记录。
- 未到期练习、直接出牌和同一到期窗口的第二次尝试都不改变 SRS。
- 重新开局不能让同一 `dueAt` 再次计分。
- 错误不会把项目重置为 L0，也不会使卡牌基础效果失效。
- L5 项目能够按维护日期重新进入候选池。
- 卡牌基础数值不读取掌握度。
- 固定 seed 和相同操作序列得到相同战斗结果。
- 敌人和遗物不能修改练习答案、计分资格或无障碍设置。
- 导入内容缺少释义时不能进入战斗，补全后可以启用。
- 同词不同义拥有独立学习状态。
- Web Speech API 不可用时仍可完成整局。

## 13. 路线图

- **v0.1 核心验证**：固定三战流程、词汇、SRS、自定义导入。
- **v0.2 Roguelike 化**：程序地图、完整牌组构筑、卡牌奖励、篝火、商店、事件和更完整遗物池。
- **v0.3 内容扩展**：初中、高中、CET6 内容包和首批语法概念；验证多练习轮换和语法迁移题。
- **v0.4 产品基础**：内容包管理、学习报告、移动端响应式优化和数据迁移工具。
- **v1.0 产品化**：账号与云同步、更多角色、可选离线插画、受控的后端富化能力。

## 14. 风险与应对

| 风险 | 设计应对 | 验证方式 |
|---|---|---|
| 过载拖慢战斗 | 每战 3 点专注、无每回合刷新、交互留在战斗界面 | 回合时长、退出位置、定性访谈 |
| 玩家只挑会的词 | 勇气奖励、挑战加成、到期内容不因刷新消失 | 到期项目尝试覆盖率 |
| 新词仍显得弱 | 战前教学后即可进行有支架过载，基础战力完全一致 | 新旧项目出牌率与过载率 |
| SRS 被重复开局刷高 | 到期窗口计分资格，不依赖 sessionId 单独防重 | 自动化状态机测试 |
| 语法变成背固定答案 | 一个概念绑定多道轮换练习，后续使用迁移题验收 | 未见句式正确率 |
| 多义词掌握度失真 | 一个目标词义一个 VocabularyUnit | 内容校验与状态隔离测试 |
| 自定义数据质量差 | 预览、错误报告、pending 状态、未完成内容禁用 | 导入夹具与可用率统计 |
| 词典或词频版权不清 | 保存来源和许可，不把 COCA 等名称视为天然可商用 | 发布前数据许可审查 |
| 浏览器语音不稳定 | 能力检测、替代题、内置包后续可使用授权音频 | 跨浏览器测试 |
| 本地数据丢失 | 导出、版本迁移、会话快照；产品化后再做云同步 | 恢复与迁移测试 |

## 15. 最终决策清单

以下原则视为实现约束，除非重新进行设计评审，不应在实施阶段自行改回：

- 战斗卡牌与学习项目使用双层模型。
- 卡牌基础效果不读取掌握度，答错也不会哑火。
- 新内容先在安全界面教学，再进入战斗。
- 只有到期窗口内第一次符合条件的作答可以改变 SRS。
- 重新开局、重复练习和直接出牌不能刷学习进度。
- 一个词义是一个原子词汇学习项目；一个语法概念必须拥有多道轮换练习。
- 过载使用每场有限专注，不使用每回合强制刷新的 quiz 配额。
- 错误只失去本次额外收益并获得反馈，不附加生命惩罚或全进度清零。
- MVP 不做 AI 插画、LLM 富化和运行时第三方密钥调用。
- MVP 先验证固定三战流程，通过后再扩展为真正的爬塔结构。

能量、卡牌数值、专注数量、间隔天数和成功指标均是首轮实验参数；核心接口必须允许配置，但它们的调整不得破坏以上原则。
